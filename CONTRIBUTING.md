# Как развивать пакет

## Что считается полезным изменением

Подходят уточнения входа и выхода навыка, новые edge cases, исправленные примеры, новые правила приватности, точнее описанные триггеры, дополнительные форматы и визуальные карты для документации. Изменение должно помогать с конкретной задачей и добавлять проверяемое правило или артефакт.

## Как добавить новый навык

1. Выберите короткое имя папки в kebab-case.
2. Создайте `<skill-name>/SKILL.md`.
3. Добавьте YAML frontmatter с `name` и точным `description`, по которому агент сможет распознать задачу.
4. Опишите вход: обязательные и необязательные данные, поведение при недостатке данных.
5. Опишите процесс по шагам.
6. Зафиксируйте выходной контракт, лучше с шаблоном.
7. Добавьте самопроверку, edge cases, связи с остальными навыками и раздел «что навык никогда не делает».
8. Добавьте пример в `docs/examples/` и строку в README.

## Проверка перед pull request

- [ ] путь к каждому `SKILL.md` работает из README
- [ ] описание навыка содержит реальные триггеры и понятный предмет работы
- [ ] вход и выход можно проверить на одном небольшом примере
- [ ] нет фальшивых цифр, кейсов и ссылок
- [ ] нет закрытых персональных данных, токенов и скриншотов переписок
- [ ] русский текст проходит редакторскую проверку на штампы, искусственные противопоставления и пустые усилители
- [ ] изображения имеют белый или тёплый белый фон и не обрезают подписи

## Обновление существующего навыка

Сначала определите, меняется ли поведение или только пояснение. Если меняется поведение, дополните edge case и пример. Если обновляется входной контракт, проверьте связи с остальными навыками. Не убирайте существующее ограничение, пока не появится более точное правило.

## Стиль документации

Пишите конкретными формулировками. Вместо «помогает улучшить контент» укажите, какой вход берётся, какое решение принимается и какой артефакт появляется. Для длинных текстов проводите второй редакторский проход через `metaflora-humanizer`.
